### Phase 4 关键架构决策

1. **职责拆分清晰**：resolve_products 只管产品**身份**；qty/price 对齐由独立的 parse_contract_items 节点负责（plan v1.1 P1-A）。避免 generate_contract 直接读 state.items 但没人写。

3. **跨轮跳过策略 3 层**（v1.9 / v1.12 / v1.13）：

5. **多组候选要求 id=N 精确选**（v1.10 P2）：单组候选允许"选 N"/名字；多组（两个产品都有歧义）必须用 `id=N`，避免裸编号同时套到所有组。

### Phase 5 关键架构决策

1. **canonical_payload 模式**：所有写子图（adjust_price/adjust_stock/voucher）的 preview 把 `{tool_name, args, preview_text}` 完整写进 PendingAction.payload。commit 节点严格从 `state.confirmed_payload` 取参数，不依赖当前 state.customer/product（多 pending / 重启 / checkpoint 恢复都能找对参数）。

2. **三档 idempotency 强度**：
   - adjust_price：无 idempotency_key（同客户产品调价多次允许）
   - adjust_stock：`stk:{conv}:{user}:{product}:{delta}` 5 min 同 context 内复用
   - voucher：`vch:{order_id}:{voucher_type}` 12 小时**全局**强幂等（跨 context 命中 fail closed）

3. **voucher_type 强解析**（v1.4 P1-B）：从 `state.voucher_type` → `extracted_hints` → keyword（"出库"/"入库"/"发货"/"收货"）多级解析；解析失败时 ask_user 不创建 pending。idempotency_key 含 voucher_type 防漏掉同订单不同类型。

4. **跨 context 幂等 fail closed**（v1.3 P1-B）：B 在群聊发起的 voucher 请求，如果 A 已在私聊创建过同订单 pending，B 必须 fail closed（不能拿到 A 的 action_id），用户看到"已有凭证申请待确认/处理中"。

5. **fallback 协议两档**（spec §12.1）：
   - 写 tool path strict 400 / max_retries 耗尽 → `LLMFallbackError` + `metrics.incr("llm.fallback")`
   - 读 tool path strict 400 → 原样 raise `httpx.HTTPStatusError`（让上层 caller 决定降级）

6. **PendingAction.idempotency_key 字段**：本 phase 加，便于回执端 inspect / 日志 / 调试；不影响存储格式（只是从 redis hash 读出来）。
