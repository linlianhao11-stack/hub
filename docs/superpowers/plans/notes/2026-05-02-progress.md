### Phase 4 关键架构决策

1. **职责拆分清晰**：resolve_products 只管产品**身份**；qty/price 对齐由独立的 parse_contract_items 节点负责（plan v1.1 P1-A）。避免 generate_contract 直接读 state.items 但没人写。

2. **extract_contract_context 在子图入口**（v1.8 P1-A+B）：set_origin 后第一个，一次抽完用户原文里 customer_name + product_hints + items_raw + shipping，全部写 state。这样多候选 ask_user 跳转之前所有 hints 已落库，跨轮短消息（"选 2"）跳过 LLM 不覆盖。

3. **跨轮跳过策略 3 层**（v1.9 / v1.12 / v1.13）：
   - `_looks_like_pure_selection`: 纯数字、"选 N"、单/多 id=N、action_id 前缀、确认词
   - `_looks_like_candidate_id_reference`: candidate_products 非空 + 消息含至少一个有效候选 id
   - 其他短消息（"北京海淀"/"13800001111" 无候选）仍走 LLM，靠 only-write-non-null 保护

4. **multi-match 绝不默认取 [0]**（plan v1.1 P1-B）：合同/报价对外文件，错客户比反问严重得多。multi 命中时写 `candidate_customers` / `candidate_products[hint]` + missing_fields 加 `customer_choice` / `product_choice:{hint}`。

5. **多组候选要求 id=N 精确选**（v1.10 P2）：单组候选允许"选 N"/名字；多组（两个产品都有歧义）必须用 `id=N`，避免裸编号同时套到所有组。

6. **format_response → cleanup 顺序**（v1.11 P1）：generate_contract 不清状态（format_response 用 state.customer.name + len(state.items) 写回执）；cleanup_after_contract 在 format_response 之后跑。
